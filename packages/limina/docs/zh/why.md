# 为什么需要 Limina

TypeScript 单体仓库一开始通常很简单：一个根 `tsconfig.json`，几个包，再加一个类型检查脚本。

在这个阶段，代码结构比较容易理解。某个文件属于哪个包、依赖了哪些模块、由哪个配置检查、构建时应该排在什么位置，通常都可以直接判断出来。

但仓库增长后，一个包内部也会变复杂。它可能同时包含浏览器运行时代码、Node 运行时代码、共享模块、测试模块、工具脚本、构建配置和发布入口。为了让这些代码获得正确的类型环境，团队通常会为不同场景配置不同的 `tsconfig`。

这时，真正的负担不再是“写一个 `tsconfig`”，而是在多个 `tsconfig` 之间长期维护一张准确的 TypeScript 项目引用图。每一次源码导入变化，都可能影响构建顺序、运行时边界和检查覆盖范围。

Limina 解决的核心问题，就是基于 TypeScript 项目引用建立一张可验证的工程图，让复杂单体仓库中的代码结构保持可理解、可审查、可预测。

## 一个包也可能是一组工程边界

在简单项目里，包通常可以被理解成一个边界：一个包、一组源码、一个类型检查入口。

但在复杂 TypeScript 单体仓库中，一个包内部可能已经包含多组边界。例如，一个 VitePress 相关包可能同时包含：

- 面向浏览器的客户端运行时代码。
- 面向 Node 的服务端或构建时代码。
- 浏览器和 Node 都可能复用的共享模块。
- 测试代码。
- 构建脚本和工具脚本。
- 面向使用方发布的多个公开入口。

这些代码虽然位于同一个包中，但它们不应该被同一套类型环境简单覆盖。浏览器代码不应该依赖 Node 内置模块；共享模块不应该意外绑定某个特定运行时；测试代码和工具脚本也不应该污染发布入口的构建关系。

因此，拆分多个 `tsconfig` 是合理的。问题在于：拆分之后，这些配置之间的依赖关系也必须被维护。

## 项目引用增强了构建，也增加了维护成本

TypeScript 项目引用适合大型仓库。它可以让类型构建按依赖顺序执行，并复用增量构建结果。

但项目引用的前提是：引用图必须准确。

最典型的维护成本并不一定来自跨包访问，而是来自[一个包内部的多组工程边界](#一个包也可能是一组工程边界)。当某个模块通过相对路径访问同包内另一个模块时，例如：

```ts
import { resolveThemeConfig } from '../shared/theme';
```

这看起来只是一次普通的相对导入。但如果导入方和被导入方分别由不同的 `tsconfig` 管辖，那么这条源码依赖也需要反映到 TypeScript 项目引用图中。否则源码关系已经存在，构建图却没有表达这条关系。

在没有自动治理的情况下，开发者需要自己判断这次导入是否跨越了不同的 `tsconfig` 管辖范围，依赖方是否需要新增 `references`，旧的 `references` 是否还必要，以及这条关系是否破坏了运行时边界。

配置越多，这种判断越难长期依靠人工完成。项目引用虽然增强了增量构建能力，也把一部分结构维护成本转移给了开发者：真实源码依赖变化后，`references` 需要同步变化；源码依赖删除后，旧引用也需要被清理。只要这两套关系不同步，构建顺序、检查范围和运行时边界就会开始变得不可预测。

## Limina 从项目引用图开始治理

Limina 的治理不是脱离 TypeScript 的额外规则系统。它建立在 TypeScript 项目引用之上。

团队仍然维护现有源码配置和包边界。Limina 会读取这些配置和真实源码导入关系，生成可增量执行的类型构建图，并检查这张图是否和源码结构一致。

这意味着用户不需要把全部精力放在手写和同步 `references` 上。更重要的是，构建图不再只是静态配置，而是可以被生成、校验和持续运行的工程事实。

当源码结构变化时，Limina 会围绕这张工程图继续回答几个问题：真实导入是否要求新的项目引用，已有项目引用是否仍然必要，跨运行时依赖是否违反边界约束，源码文件是否属于明确的检查范围，以及生成的构建关系是否仍然适合增量执行。

后面的几个场景本质上都在解释同一件事：让代码关系进入同一张可验证的工程图，而不是让开发者在每次变更中手动推演整张 `references` 图。

## 典型场景：同一个包支持多个运行时

假设一个包同时包含 `client`、`node` 和 `shared` 三类源码：

```txt
packages/example/
  src/client/
  src/node/
  src/shared/
```

通常它们会分别有不同的类型环境：

```txt
src/client/tsconfig.json   # 浏览器运行时
src/node/tsconfig.json     # Node 运行时
src/shared/tsconfig.json   # 共享模块
```

这时，基本结构预期很清楚：`client` 可以依赖 `shared`，但不应该依赖 `node`；`shared` 应该保持可复用，不应该绑定某个具体运行时；`node` 可以使用 Node 类型和 Node 内置模块。

问题在于，这些预期需要反映到项目引用图和源码边界中。否则，开发者每新增一次跨 `tsconfig` 管辖范围的相对导入，都需要回到[项目引用增强了构建，也增加了维护成本](#项目引用增强了构建-也增加了维护成本)中描述的判断：依赖方是否应该新增 `references`，这条边是否违反运行时边界，旧的引用关系是否仍然必要。

Limina 的价值在于把这些判断变成检查。用户通过源码表达真实依赖，Limina 基于这些依赖维护可验证的工程图，并在结构不符合预期时给出失败信息。

## 典型场景：测试和工具代码不应污染主构建

复杂包中还经常包含测试代码和工具脚本：

```txt
packages/example/
  src/
  scripts/
  tests/
  vitest.config.ts
  build.config.ts
```

这些文件需要类型检查，但它们不一定应该参与生产代码的构建关系。

如果没有清晰的配置边界，测试代码可能引用仅测试可用的依赖，工具脚本可能引用 Node 专属模块，而这些关系又可能被错误地混入生产构建图。这里的问题和[同一个包支持多个运行时](#典型场景-同一个包支持多个运行时)相同：不同源码范围需要不同的类型环境，也需要明确的引用关系。

Limina 关注的不是“仓库里有多少个检查脚本”，而是这些脚本背后的源码范围是否清楚。测试文件应该只进入测试配置，工具脚本应该使用适合自己的类型环境，生产源码不应该被测试配置或工具配置意外拥有。

当这些关系可以被检查，团队就能更清楚地理解每段代码在仓库结构中的位置。

## 典型场景：代码能跑，但结构已经不可预测

跨包相对导入是单体仓库中常见的结构问题：

```ts
import { Button } from '../../ui/src/Button';
```

这段代码短期看可能可以运行，但它绕过了包名、依赖声明和公开入口。结果是：`package.json` 看不出真实依赖，包的公开入口无法约束实际使用方式，开发者还需要手动推演这条导入是否应该进入 TypeScript 项目引用图。

更可预测的方式是通过包名和公开入口访问：

```ts
import { Button } from '@acme/ui';
```

这样，代码关系会同时出现在源码导入、依赖声明、公开入口和项目图中。评审者也能清楚判断这次变更是否符合仓库结构。

`#imports` 也遵守这个思路：相对 target 是声明它的包作用域内部入口，不能借它访问其他包；package target 可以指向三方包或工作区依赖，但仍然需要由导入文件所属的源码归属方授权。

这和前文的[同包内跨 `tsconfig` 相对导入](#项目引用增强了构建也增加了维护成本)是同一类问题：代码能解析，不代表结构关系已经被正确表达。Limina 的检查本质上就是帮助团队确认这段代码在整个仓库结构中是否处于明确、合理、可预测的位置。

## 最终目标：降低结构维护成本

Limina 的目标不是让仓库增加更多规则，而是降低复杂结构带来的维护成本。

对于大型 TypeScript 单体仓库来说，真正困难的地方不是写一个 `tsconfig`，而是在多个包、多个运行时、多个检查器和多个发布入口之间长期保持一致。前面几个场景分别从运行时边界、测试与工具配置、跨包访问解释了这个问题，但它们最终都指向同一件事：源码关系需要和 TypeScript 项目引用图保持一致。

接入 Limina 后，团队可以更稳定地回答这些问题：

- 这个文件属于哪个源码范围？
- 它应该由哪个 `tsconfig` 管辖？
- 它依赖的模块是否符合运行时边界？
- 项目引用是否准确表达了真实源码依赖？
- 类型构建图是否可以增量执行？
- 测试、工具和生产源码是否被清晰分离？
- 发布前产物是否仍然符合源码阶段建立的结构预期？

当这些问题都能通过自动检查回答，开发者就不需要在每次变更中手动推演整张项目引用图。

这就是 Limina 的核心价值：在 TypeScript 项目引用的基础上建立可验证的工程图，让复杂单体仓库中的代码结构保持可理解、可审查、可预测。
