# 为什么需要 Limina

TypeScript 单体仓库往往一开始很简单：一个根 `tsconfig.json`，几个包，再加一个 `typecheck` 脚本，看起来就够了。

仓库变大后，问题通常不是某个命令完全不能跑，而是不同工具开始看见不同的“事实”。编辑器关心快速拿到本地类型，`tsc -b` 需要干净的项目引用图，Vue 或 Svelte 文件需要框架检查器，包之间通过声明过的依赖互相访问，发布时消费者安装到的又是构建后的产物。

这些事情有关联，但 TypeScript 不会自动证明它们彼此一致。Limina 解决的就是这个空隙。

> Limina 是 TypeScript monorepo 的架构治理 CLI：让源码导入、项目引用、包边界、导出的依赖图事实和发布产物描述同一组事实。

## Limina 工作在哪一层

Limina 不替你打包，不替你运行测试，也不负责调度仓库里的所有任务。它回答的是另一层问题：这个仓库的结构是否可信？

也就是说，Limina 会检查 TypeScript 源码依赖图、`package.json` 依赖和导出、项目引用、检查器覆盖、运行时边界、限定范围内可导出的 source/artifact 图事实以及发布产物是否对得上。它可以和现有工作流一起使用：

```json [package.json]
{
  "scripts": {
    "typecheck": "limina check",
    "prepublishOnly": "limina package check && limina release check"
  }
}
```

这里现有工作流继续处理普通执行问题；Limina 负责证明这些流程依赖的项目关系没有漂移，并在需要时导出受管辖域限定的依赖图。

::: tip
如果你想看更具体的失败场景，可以继续阅读[架构一致性](./architecture-conformance.md)。如果你想把 Limina 接入仓库，可以从[快速开始](./getting-started.md)入手。
:::

## project references 很有用，但很容易漂移

TypeScript 的 `tsc --build` 适合大型仓库。它可以根据 project references 理解项目之间的依赖关系，按正确顺序构建，并复用增量构建信息。

现实里的难点是维护成本。每个可构建项目都要有合适的 `tsconfig`，被引用项目要满足 `composite` 和声明文件输出要求，项目之间新增或删除依赖后，`references` 也要及时同步。仓库里如果同时存在 lib、test、storybook、tools 等不同场景，还会出现多张项目图。靠人工长期维护这些声明边，很容易让构建图和真实源码关系分开。

Limina 的做法不是替代 TypeScript，而是接管这张用于构建的声明图。你提供普通源码 `tsconfig` 作为检查器入口；Limina 会从源码导入、TypeScript 模块解析和 `tsconfig` 管辖范围里推导真实依赖，再生成并校验声明构建所需的类型输出图。

换句话说，用户维护的是源码配置和包边界，Limina 维护的是生成 dts 图。`graph:check` 每次都会先 prepare 这张图，再校验它的结构选项、源码伴随关系、导出解析、条件域、架构规则和源码 / 产物依赖语义。

## 多种检查器也要进入同一张事实图

很多仓库并不只有 `.ts` 文件。Vue、Svelte、原生 TypeScript、不同运行时或不同测试环境，往往需要不同检查器理解各自的源码。

如果这些检查器只是各跑各的命令，仓库仍然缺少一层共同事实：哪些文件被谁覆盖，哪些源码能参与声明构建，哪些只能做类型诊断，哪些文件不该同时被多个同类检查器管辖。

Limina 会把检查器当作项目图的一部分。每个检查器提供自己的文件扩展、模块解析和执行方式；能稳定进入构建模式的检查器会消费生成 dts 图，只能直接做类型诊断的检查器仍然参与源码覆盖和图建模。这样框架文件不会被排除在架构治理之外，也不会悄悄绕过 TypeScript 项目边界。

## Limina 主要守住哪些关系

在单体仓库里，最容易出问题的不是某一份配置，而是几份配置之间说法不一致。Limina 把这些关系放在一起检查：

| 容易漂移的关系               | 漂移后的表现                                        | Limina 做什么                                          |
| ---------------------------- | --------------------------------------------------- | ------------------------------------------------------ |
| 源码导入和声明图             | 代码依赖已经变化，但人工维护的构建引用难以及时同步  | 从源码事实生成 dts 图，并校验生成图的结构和语义        |
| 包访问和 `package.json`      | 跨包相对导入、裸包依赖没声明、`#imports` 越界       | 让跨包访问经过包名、依赖声明和公开入口                 |
| 依赖声明和构建产物           | 有的导出指向源码，有的导出指向 `dist`，语义混在一起 | 先解析公开导出，再判断应该走声明图还是导出的产物边     |
| 源码文件和检查器覆盖         | 文件没人检查，或者同一文件被多个检查器重复拥有      | 验证不同检查器的覆盖范围、声明归属和源码边界           |
| 源码通过检查和发布产物可用性 | 源码能过类型检查，但 `dist` 里的 `types` 或入口坏了 | 在构建后检查消费者真正安装到的包、运行时边界和发布卫生 |

这些检查的目标不是增加规则感，而是减少“看起来能跑，实际已经不一致”的隐性风险。

## 工作区依赖需要明确语义

包依赖声明只表示当前包被允许访问另一个包，但它没有说明你消费的是源码还是构建产物。真正决定语义的是被依赖包的 `package.json#exports` 解析结果。

Limina 会先解析公开导出，再决定这个导入在仓库里意味着什么。它不只看字符串配置是否存在，还会同时确认类型解析和运行时解析是否都能到达合理入口。纯类型入口可以只落到声明文件；但如果 TypeScript 在声明上下文里只能解析到运行时 JavaScript，或者运行时解析根本找不到公开入口，这个导出就不应该被当作可靠事实继续使用。

例如 `@acme/app` 依赖 `@acme/core`：

```json [packages/app/package.json]
{
  "dependencies": {
    "@acme/core": "workspace:*"
  }
}
```

如果 `@acme/core` 的公开入口解析到源码：

```json [packages/core/package.json]
{
  "exports": {
    ".": "./src/index.ts"
  }
}
```

那么 `app` 导入 `@acme/core` 时，Limina 会把它视为源码依赖，并在生成 dts 图时把 `app` 的类型输出项目连到 `core` 的类型输出项目。后续 `graph:check` 校验的是这张生成图是否满足声明构建、条件域、包依赖和架构规则。

如果另一个公开入口解析到构建产物：

```json [packages/core/package.json]
{
  "exports": {
    "./runtime": {
      "types": "./dist/runtime.d.ts",
      "import": "./dist/runtime.js"
    }
  }
}
```

那么消费方不是在依赖 `core` 的源码图，而是在依赖 `core` 的构建输出。此时 Limina 不要求项目引用，而是在依赖图导出中按导入方 tsconfig 的条件域记录这条产物边。

::: tip
依赖图导出尊重 Limina 管辖的 tsconfig 域和各自的 `compilerOptions.customConditions`。它适合用于审查和诊断，不是全局构建解析图，也不是任务编排保证。
:::

## 运行时边界应该能被证明

单体仓库经常会同时包含浏览器、Node、共享代码、构建脚本和测试环境。仅靠目录命名或代码评审，很难长期保证它们不会互相穿透。

Limina 的图规则把这类边界放到真实导入和生成图上验证。你可以给某个源码配置标记规则，再禁止它访问特定声明边界、包、`#imports` 或 Node 内置模块。规则命中时，诊断会指向具体导入文件或生成图边，并显示配置里的原因。

它还会检查声明引用树里的 `customConditions` 是否一致。也就是说，当某一棵声明图代表特定的包解析条件或运行时条件时，Limina 会防止它在引用下游项目时混入另一套条件语义。

## 包边界应该清楚

单体仓库里最常见的边界逃逸，是跨包相对导入：

```ts
import { Button } from '../../ui/src/Button';
```

这行代码绕过了包名、依赖声明和公开导出。短期看它能工作，长期看会让几个系统产生不同理解：`package.json` 看不出真实依赖，`exports` 约束不了公开 API，发布产物可能没有对应入口，内部目录重构也容易破坏别的包。

Limina 会把这类导入报告出来，推动跨包访问统一走包名和公开入口：

```ts
import { Button } from '@acme/ui';
```

它也会检查裸包导入是否写在最近的 `package.json` 里，`#imports` 是否匹配当前包的 `imports` 字段，以及非聚合 `tsconfig` 是否混入了多个包的源码。

这些规则让评审者不必沿着相对路径追踪依赖关系。包之间的关系应该出现在清单文件、导出入口和项目图里，而不是藏在某个 `../../` 后面。

## 检查通过不等于发布可用

源码图通过检查之后，发布产物仍然可能坏。消费者安装的是打包后的 npm tarball，不是仓库里的源码 `tsconfig`。

Limina 的 package 检查会在构建后验证面向消费者的包：`exports` 能不能解析，`types` 是否存在，运行时导入是否合理，依赖声明是否完整，包内自引用是否真的由输出包导出。release 检查会继续验证打包清单、README、license、误带文件、缺失文件、source map、本地协议泄漏，以及工作区依赖相对已发布基线是否可安全发布。

例如源码类型检查可以通过，但 `dist/package.json` 的 `types` 指向了不存在的声明文件，或者浏览器产物里还导入了 `node:fs`。这类问题应该在发布前被发现，而不是等消费者安装后才暴露。

## 最终得到什么

Limina 的目标不是让仓库多一层复杂配置，而是把已经存在的复杂关系变得可见、可审查、可运行。

接入之后，你得到的是一张更可信的 TypeScript monorepo 项目图：源码依赖能推导，项目引用能生成和校验，跨包访问有明确边界，检查器覆盖不会悄悄漏掉文件，导出的产物边能和真实产物消费对齐，发布前也能验证消费者实际拿到的包。

当这些关系都能被 CI 检查，架构漂移就不会只靠代码审查时的经验判断。它会变成普通 PR 里可以讨论、可以修复、可以持续维护的工程事实。
