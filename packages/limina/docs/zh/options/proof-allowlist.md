# Proof allowlist

Allowlist entries 是显式源码覆盖例外。

```js
import { defineConfig } from 'limina';

export default defineConfig({
  proof: {
    allowlist: [
      {
        file: 'src/generated/runtime.d.ts',
        reason: 'Generated by the runtime build step.',
      },
    ],
  },
});
```

## `file`

`file` 是允许例外的源码文件路径，相对 workspace root。它必须指向具体文件，不应该用模糊 glob 扩大例外范围。

## `reason`

`reason` 解释这个文件为什么可以不走普通 checker coverage。它不能为空，因为 allowlist 是给 reviewer 看的架构例外，不只是让 CI 变绿的开关。

例如某个声明文件只由构建步骤生成：

```ts
// src/generated/runtime.d.ts
declare const runtimeVersion: string;
```

如果它落在 `config.source.include` 范围内，却不属于任何 checker entry，`limina proof check` 会要求你处理它。加到 `proof.allowlist` 后，检查不会再把它当作遗漏，但 reason 会留在配置里，后续 review 能看出这个例外为什么存在。

完整一点看，目录可以是：

```text
packages/core/
  src/index.ts
  src/generated/runtime.d.ts
  tsconfig.lib.dts.json
  tsconfig.lib.json
```

`src/generated/runtime.d.ts` 在 `config.source.include` 范围内，但它不是由本仓库源码维护，而是构建步骤写出的声明文件。运行 `pnpm exec limina proof check` 时，Limina 会发现这个文件没有被 checker entry 覆盖，于是把它报告为 uncovered source。

把它加入 allowlist 后：

```js
proof: {
  allowlist: [
    {
      file: 'packages/core/src/generated/runtime.d.ts',
      reason: 'Generated by the runtime build step.',
    },
  ],
}
```

结果是 proof check 通过这个例外，同时保留 `reason` 给 reviewer 判断它是否仍然合理。这里的重点是 allowlist 只解释少量有意例外，不用来批量隐藏普通源码。
