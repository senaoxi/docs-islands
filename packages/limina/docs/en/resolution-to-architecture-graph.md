# From Resolution to Architecture Graph

Limina does not need a special module resolver.

For Limina, a general resolver has usually done its job once it can resolve an `import` to a real file under the current project configuration.

A resolver answers:

```text
which file does this import point at?
```

Limina continues with:

```text
what does that file mean in this monorepo architecture?
```

That is the core of Limina's value.

## Resolution Is Only the First Step

Suppose source code imports another workspace package:

```ts
import { createClient } from '@acme/core';
```

The resolver only needs to produce a file target:

```text
@acme/core
  -> packages/core/src/index.ts
```

At that point, module resolution is complete. Limina still needs to ask:

```text
packages/core/src/index.ts
  -> which package owns it?
  -> which source type config owns it?
  -> is it governed by another project?
  -> should it map to a type-output module?
  -> should it generate a reference?
  -> does it violate graph rules?
```

So Limina does not require a resolver to understand TypeScript `references`. It only requires an accurate file target. Limina owns the reference decision.

## General Resolvers Are Usually Enough

Limina uses Oxc Resolver, but not because Oxc Resolver understands TypeScript project references.

Its value is lower level:

```text
import specifier -> resolved file
```

Limina handles the later steps:

```text
resolved file -> owning source type config -> type-output module
```

The full chain is:

```text
import specifier
  -> resolved file
  -> owning source type config
  -> type-output module
```

Accuracy comes from Limina's project model:

1. Users declare source type config entries.
2. Limina resolves the file set governed by each entry.
3. Each governed source file can belong to a source type config.
4. Each source type config has a matching type-output module.
5. Cross-project source imports can map to references between type-output modules.

In this model, the resolver only needs to solve "import to file".

## Successful Resolution Is Not Legality

In ordinary build flows, "it resolved" often means the build can continue.

In Limina, resolution is evidence, not the conclusion.

For example:

```ts
import '@acme/core/internal';
```

If it resolves to:

```text
packages/core/src/internal.ts
```

that does not make the import valid. It may mean the current package bypassed `@acme/core`'s public entry.

Or:

```ts
import '@acme/core/runtime';
```

If it resolves to:

```text
packages/core/dist/runtime.d.ts
```

Limina should not force that into a source `references` relationship. It is artifact consumption and may need to appear as an exported artifact edge instead.

Or:

```ts
import 'node:fs';
```

Even if the resolver recognizes a Node built-in, that does not make it legal in a browser entry.

Limina's chain is not:

```text
resolves -> legal
```

It is:

```text
resolves
  -> determine package ownership
  -> determine source or artifact ownership
  -> apply graph rules, package boundaries, runtime boundaries, and graph-export semantics
  -> pass or fail
```

## Artifacts Must Not Be Quietly Rewritten to Source

If an import resolves to:

```text
packages/core/dist/index.d.ts
```

Limina should not walk backward to:

```text
packages/core/src/index.ts
```

just to make the source graph look more complete.

The resolution result already says that the public entry exposes built output. That edge should be treated as artifact consumption, not as a source project reference.

This matters because Limina preserves the actual contract exposed through `package.json#exports`. If a package exposes `dist`, Limina should not pretend it exposed `src`; otherwise it hides mismatches between the package entry, build output, and type declarations.

## Invisible Edges Must Be Declared

Not every real dependency can be found through static imports.

Examples include:

- imports that only appear in generated files;
- route, plugin, or command tables generated by build plugins;
- modules registered at runtime through manifests;
- framework macros or compiler plugins that create dependencies after transformation;
- virtual modules mapped to real source during build;
- DI containers connected by string tokens.

These edges are real, but the static import graph cannot prove them.

They should not be written into ordinary source tsconfigs as TypeScript `references`. The better fit is Limina's dedicated field:

```json
{
  "liminaOptions": {
    "implicitRefs": [
      {
        "path": "../core/tsconfig.lib.json",
        "reason": "This dependency is connected by a generated route manifest; source has no static import."
      }
    ]
  }
}
```

`implicitRefs` means:

> This is an implicit dependency edge explicitly declared by the user.
> "Implicit" means invisible to the static source import graph, not secretly added by Limina.

It participates only in Limina's references graph and does not change TypeScript's native interpretation of the source tsconfig.

## Limina's Resolution Model

The full flow is:

```text
source import
  -> static import record
  -> resolver target
  -> package ownership
  -> source or artifact ownership
  -> type-output module or exported artifact edge
  -> graph rules, package boundaries, and runtime boundaries
  -> generated graph or diagnostic
```

Resolvers are good at the middle step:

```text
import specifier -> file
```

Limina's core is the step after it:

```text
file -> architectural meaning
```

## Different Results Mean Different Things

The same successful resolution can produce different architecture meanings:

| Resolution result              | Limina meaning                        |
| ------------------------------ | ------------------------------------- |
| Another governed source file   | May generate `references`             |
| Another package's built output | May become an exported artifact edge  |
| External package entry         | Check dependency declaration          |
| Current package internal file  | Check it stays inside the package     |
| Another package internal file  | Possible public-entry bypass          |
| Node built-in module           | Check runtime boundary                |
| Unresolvable specifier         | Entry, path, or configuration problem |

Limina never treats "can resolve" as "is legal".

It cares about:

```text
what architecture object owns the resolved file?
which rule explains this edge?
can the edge be reviewed and reproduced?
```

## Diagnostics Must Point Back to the Import

There is another reason Limina should not quietly rewrite resolver results: diagnostics need to be reviewable.

When an edge is wrong, Limina needs to tell the user:

```text
which file?
which line?
which import form?
which specifier?
where did it resolve?
why does that result violate the rule?
should the fix touch source, package exports, type config, or implicitRefs?
```

If a tool rewrites artifacts back to source, treats dynamic dependencies as static imports, or accepts internal paths as public entries to make the graph look nicer, its diagnostics stop being trustworthy.

The important property of an architecture governance tool is not guessing as much as possible. It is:

```text
state the provable facts clearly
expose what cannot be proven
make the user repair the structure explicitly
```

## Mental Model

Do not think of Limina as:

```text
a stronger module resolver
```

A better model is:

```text
a monorepo architecture governor that uses module resolution as evidence
```

The resolver owns:

```text
import specifier -> file
```

Limina owns:

```text
file -> package ownership
file -> source type config ownership
source type config -> type-output module
import relationship -> references / artifact edge / package dependency / boundary diagnostic
```

That is why Limina does not treat Node resolution, TypeScript resolution, Oxc Resolver, bundler resolution, or Knip as the whole answer. They can all provide facts. The final interpretation belongs to Limina's monorepo graph model.

## Conclusion

General module resolvers are usually enough for Limina because Limina does not require them to understand TypeScript `references` directly.

The resolver only needs to answer:

```text
which file does this import point at?
```

Limina continues with:

```text
which package owns this file?
which source type config owns this file?
which type-output module matches that source type config?
should this relationship generate references, an artifact edge, a package dependency check, or a boundary error?
```

So Limina's core is not inventing a more specialized resolver. It is establishing a monorepo graph model that can interpret resolver results.

More precisely:

> Resolvers provide file facts.
> Limina interprets those facts and turns them into checkable, buildable, fail-and-fix architecture graph relationships.
